## 1. Problem statement

`SessionContext` currently holds the graph editor's selection state as
direct reactive fields:

```python
# packages/haywire-core/src/haywire/ui/context.py
class SessionContext:
    active_graph: Reactive[Optional["BaseGraph"]] = reactive_field(None)
    active_node: Reactive[Optional["NodeWrapper"]] = reactive_field(None)
    active_edge: Reactive[Optional["EdgeWrapper"]] = reactive_field(None)
    active_port: Reactive[Optional["DataPort"]] = reactive_field(None)
    selected_nodes: Reactive[Set[str]] = reactive_field(set())
    selected_edges: Reactive[Set[str]] = reactive_field(set())
    active_graph_path: Reactive[Optional[Any]] = reactive_field(None)
    clipboard: Reactive[Optional["ClipboardData"]] = reactive_field(None)
    # ... plus non-graph-editor fields (active_file, active_library, themes, ...)
```

Two things are wrong with this:

1. **Layering.** `SessionContext` lives in `haywire-core` (the framework).
   The graph editor lives in `haybale-studio` (a library, since commit
   `998a661`). Editor-specific state in framework-level objects creates
   an inverted dependency: framework code carries fields that only the
   editor library uses, and the editor library writes to framework
   fields it doesn't own.

2. **Inconsistency with v1.1.** v1.1 established the rule **scope = base
   class** for per-session library runtime state: a library declares a
   `SessionState` subclass and the container routes per-session
   instances. The graph editor's selection state is exactly this kind of
   per-session library runtime state — but it predates v1.1 and lives
   on a parallel mechanism (direct `SessionContext` fields). Two
   mechanisms for the same concept.

v1.2 collapses both: introduce `EditState(SessionState)` in
`haybale-studio`, move 8 fields into it, remove them from
`SessionContext`, rewire ~97 read sites and ~30 write sites.

**This is a refactor, not a feature.** Zero observable behavior change.
Every field starts at the same default, transitions through the same
states, fires the same reactive subscriptions. Only the access path
changes: `ctx.active_node.value` → `ctx.data[EditState].active_node.value`.

## 2. Goals & non-goals

### Goals

- `EditState` lives in `barn/haybale-studio/haybale_studio/state/edit_state.py`.
- The 8 fields (`active_graph`, `active_node`, `active_edge`,
  `active_port`, `selected_nodes`, `selected_edges`, `active_graph_path`,
  `clipboard`) move from `SessionContext` to `EditState`.
- All read and write sites migrate to `ctx.data[EditState].*`.
- `SessionContext` no longer declares the 8 fields.
- Tests pass at every commit; PR is atomic in outcome.
- Running-app smoke confirms behavior unchanged.

### Non-goals

- No Phase 2 reactive auto-tracking. Reads stay at `.value`. Phase 2 is
  a uniform sweep across all `Reactive[T]` reads, not an EditState
  concern.
- No new spec doc. `session_state.md` §9 already names `EditState` and
  the 7 fields; v1.2's scope decisions go inline in this plan.
- No additional scopes (e.g., `GraphState`, per-editor-instance state).
  v1.1 §9 Q3 settled this as v1.3+ territory.
- No move of `ClipboardData`. It stays in
  `haywire.core.undo.actions.graph_actions` (used by core's undo
  system); `EditState` imports it. See §6.2 for the documented retreat.
- No special-casing on `SessionContext` (no `ctx.edit_state` shortcut,
  no proxy fields). Pure library-state model: `ctx.data[EditState]`.
- No external-author migration note. `session_state.md` §6.3 confirmed
  zero external library consumers; this is unchanged for v1.2.

## 3. Architectural decisions

These extend `session_state.md` §9 with the resolutions from the v1.2
inquisition. They live in this plan rather than a separate spec because
the v1.1 spec already provides the conceptual frame.

### 3.1 EditState is studio-owned, not framework-owned

`session_state.md` §8.2 originally framed `EditState` as the framework's
*own* built-in `SessionState`. v1.2 inverts this: `EditState` lives in
`barn/haybale-studio/`, the same library that owns the graph editor,
the panels that read selection state, and every writer that mutates it
(verified: `grep` for writers on the 8 fields shows zero hits in
`packages/haywire-core/`).

This means:

- `EditState` registers via the standard library bootstrap path
  (`add_folder_to_registry(state/, LibraryStateRegistry)`), no
  framework-level special case.
- `EditState` participates in hot-reload like any other library state
  (changing `edit_state.py` reloads it across all sessions).
- `haywire-core` does not import `EditState`. Code in core that
  conceptually wants to read selection state cannot — but a fresh sweep
  shows core doesn't actually do this today (`panel.py:35` is a
  docstring example, `descriptor.py:4` is a comment, both rewritten in
  C5).

### 3.2 The 8 fields disappear from SessionContext

Reading `ctx.active_node` after v1.2 is `AttributeError`. There is no
proxy, no shortcut, no compatibility shim. Every call site changes to
`ctx.data[EditState].active_node.value`.

This is the only choice consistent with §3.1: if SessionContext kept the
fields as a proxy/mirror over EditState, the layering problem isn't
solved (SessionContext still surfaces editor concerns), and we'd ship a
dual-source-of-truth system that's strictly worse than today's.

### 3.3 EditState owns 8 fields, not 7

`session_state.md` §8.2 listed 7 fields. v1.2 adds an 8th: `clipboard`.

Rationale: `ClipboardData` is graph-editor-specific (it holds copied
nodes/edges via `from haywire.core.undo.actions.graph_actions import
ClipboardData`). Keeping `clipboard` on `SessionContext` while the
selection cluster moves to `EditState` would split the editor's data
model across two locations. The fields it needs to act on
(`selected_nodes`, `selected_edges`) live in `EditState`; the result of
acting on them (`clipboard`) belongs there too.

The non-editor fields stay on `SessionContext`: `active_file`,
`active_library`, `active_component`, `active_workbench_theme_key`,
`active_node_theme_key`. These are session-level, not editor-level.

### 3.4 ClipboardData stays in core

`ClipboardData` is referenced by `haywire.core.undo.actions.graph_actions`
(the undo action that records a paste). Moving the type to studio would
require moving the undo action too, which is out of scope. The class
stays in core; `EditState` imports it.

**Documented retreat.** If, during implementation, `EditState`'s import
of `ClipboardData` introduces cross-package import-cycle friction (e.g.,
because `graph_actions.py` ends up importing something that transitively
imports `EditState`), the fallback is: drop `clipboard` from EditState's
field set, leave it on `SessionContext`. The 7 selection fields still
move. This is a one-line retreat in `EditState`'s class definition and a
re-add of the `clipboard` declaration on `SessionContext`. No other plan
changes.

### 3.5 Test fixtures use real LibraryStateContainer

Tests that need to read EditState construct one via the same container
machinery the app uses, not via test-only injection methods. A
`register_edit_state(container)` fixture helper performs the
registration, then `container.attach_session(sid)` instantiates it.
This keeps tests honest about the system being tested.

The container's current API is:

```python
container.attach_session(sid)        # public, used by SessionManager
container._add_session_class(cls, registry_key)  # private, used by registry subscriber
```

The fixture needs to either (a) call `_add_session_class` directly
(name-mangled-but-not-private), or (b) we add a thin public
`register_class(cls)` method to `LibraryStateContainer`. Decision in C1
when the actual fixture is written; (a) is acceptable for tests since
the alternative is to spin up the full registry + folder scan.

## 5. EditState class — final shape

```python
# barn/haybale-studio/haybale_studio/state/edit_state.py
"""EditState — per-session graph-editor selection and clipboard state.

Owns the reactive cluster previously held on SessionContext. Migrated
out of haywire-core in v1.2 because (a) it is editor-specific, not
framework-level, and (b) it should participate in the v1.1 SessionState
mechanism like any other per-session library runtime state.

Read paths:
    edit = ctx.data[EditState]
    if edit.active_node.value is not None:
        ...

Write paths (only in haybale-studio editors and canvas handlers):
    ctx.data[EditState].active_node.value = wrapper

See internals/documentation/architecture/session_state.md.
"""

from __future__ import annotations

from typing import Any, Optional, Set, TYPE_CHECKING

from haywire.core.state import SessionState
from haywire.ui.reactive import Reactive, reactive_field

if TYPE_CHECKING:
    from haywire.core.edge.edge_wrapper import EdgeWrapper
    from haywire.core.graph.base import BaseGraph
    from haywire.core.node.base import DataPort
    from haywire.core.node.node_wrapper import NodeWrapper
    from haywire.core.undo.actions.graph_actions import ClipboardData


class EditState(SessionState):
    """Per-session graph-editor state: selection, active items, clipboard."""

    active_graph: Reactive[Optional["BaseGraph"]] = reactive_field(None)
    active_graph_path: Reactive[Optional[Any]] = reactive_field(None)

    active_node: Reactive[Optional["NodeWrapper"]] = reactive_field(None)
    active_edge: Reactive[Optional["EdgeWrapper"]] = reactive_field(None)
    active_port: Reactive[Optional["DataPort"]] = reactive_field(None)

    selected_nodes: Reactive[Set[str]] = reactive_field(set())
    selected_edges: Reactive[Set[str]] = reactive_field(set())

    clipboard: Reactive[Optional["ClipboardData"]] = reactive_field(None)
```

No `on_enable` / `on_disable` hooks needed — the reactive cells default
to None/empty per `reactive_field()`, identical to today's
`SessionContext` behavior. Writers populate values when the user opens a
file or selects a node.

The class is ~30 lines including imports and docstring. No new
abstractions.

### C2 — Dual-write window

**Goal:** Every writer of the 8 fields writes to *both* SessionContext
and EditState. Readers still read from SessionContext. Both stay in
sync.

**Why dual-write:** Atomic switching of writers and readers in one
commit makes the diff hard to review and bisect. Splitting them lets us
verify writers in isolation (C2 → no behavior change because readers
are unchanged) and readers in isolation (C3 → no behavior change because
writers populate both sides).

**Files (writers):**
- `barn/haybale-studio/haybale_studio/editors/graph_editor.py`
- `barn/haybale-studio/haybale_studio/editors/haystack_editor.py`

- `barn/haybale-studio/haybale_studio/editors/graph_canvas/handlers/selection.py`

**Pattern at each write site:**


## 8. Risks and mitigations

| Risk | Mitigation |
| --- | --- |
| **A writer is missed in C2.** Some path mutates a SessionContext field but isn't migrated to dual-write. After C5 (when the SessionContext field is removed), that path crashes with `AttributeError`. | The exhaustive sweep in §4.2 lists every writer found. The PR description requires the reviewer to re-grep against the latest `main` to confirm completeness. The smoke test in §6 C5 exercises every writer path. |
| **A reader is missed in C3 or C4.** After C5 the read crashes. | Same grep gate as above. C5's grep is the final acceptance test. |
| **`ClipboardData` import-cycle friction.** EditState (in studio) imports `ClipboardData` (in core); if core somehow imports something from studio transitively, we get a cycle. | Documented retreat in §3.4: drop `clipboard` from EditState's field set, leave it on `SessionContext`. One-line change. Detected at C1 if it would happen (the `from haywire.core.undo.actions.graph_actions import ClipboardData` either succeeds or fails when the test suite runs). |
| **Hot-reload of `edit_state.py` loses selection state mid-session.** Per `session_state.md` §3.4, `CLASS_RELOADED` re-instantiates per-session state, dropping field values. | This matches the documented v1.1 behavior for any SessionState. Editing `edit_state.py` is a developer action, not an end-user action; resetting selection is acceptable. Document in C1. |
| **Tests that mock SessionContext stop working.** Some tests construct a `Mock(spec=SessionContext)` and access `.active_node`. After C5, `spec` strips the attribute. | Migrate those tests to use the real `SessionContext` + the C1 `register_edit_state` fixture. Identified during C4. |
| **A new writer or reader lands on `main` during the migration.** Rebase introduces a site that bypasses both `EditState` and the C2 dual-write. | Rebase frequently (the migration is small enough to land in days, not weeks). The C5 grep gate catches any straggler. If `main` is moving fast, do C2–C5 in a tight sequence and don't sit on the branch. |
| **The grep gate has a false negative.** A read uses an indirect access (e.g., `getattr(ctx, "active_node")`). | Augment the grep with `getattr.*ctx.*active_\|getattr.*ctx.*selected_`. Spot-check during C5. Indirect accesses are rare in this codebase (verified during the inquisition sweep). |

## 9. Out of scope (explicit non-list)

The following are **not** part of v1.2 and should not be bundled:

- Phase 2 reactive auto-tracking. Separate effort, separate PR, applies
  uniformly to all `Reactive[T]` reads.
- Moving `ClipboardData` to studio. Requires moving the undo action,
  out of scope.
- Adding a `GraphState` or per-editor-instance scope. v1.3+ territory
  per `session_state.md` §9.
- Renaming `LibraryStateContainer` to reflect it now holds two scopes.
  v1.1 left the name; v1.2 doesn't change it.
- Refactoring `SessionContext` further (e.g., splitting `active_file` /
  `active_library` / themes into separate SessionStates). v1.2 only
  touches the editor cluster; the rest of SessionContext stays put.
- Updating `barn/haybale-testing/` panels to do anything beyond the
  mechanical migration. The panels there are deliberately
  test-fixtures-as-real-code; they migrate to read from `EditState` but
  don't gain new functionality.
