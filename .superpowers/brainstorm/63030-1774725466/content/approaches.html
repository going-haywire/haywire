Settings UI Dev Harness — 3 Approaches

How should the isolated test environment be structured?

A

Standalone NiceGUI Mini-App

A separate dev_harness.py that boots a minimal NiceGUI server rendering only the settings panel in isolation — no graph, no app shell, no library loading. You point it at any NodeSettings or LibrarySettings class and it renders the panel directly.

Pros

  • Instant startup (<2s vs ~10s for full app)
  • No graph setup or node instantiation needed
  • Easy for agent to drive via playwright / browser-use
  • Composable — swap in any settings class

Cons

  • Needs a small amount of DI wiring to inject a registry
  • Mirror resolution requires a real SettingsRegistry to be set up
B

pytest + Playwright Headless Tests

Use pytest-nicegui (or a custom fixture) to run NiceGUI in a headless browser, assert on rendered DOM elements and interactions. Tests live alongside unit tests and run in CI.

Pros

  • Fully automated and repeatable
  • Assertions are code — no human in the loop
  • CI-friendly

Cons

  • Headless — you can't see the UI while developing
  • Playwright setup overhead
  • Test failures are DOM assertions, not visual inspection
  • Does not help with visual iteration speed
C

Harness App + Agent Test Script (Recommended)

Combine A and B: a mini NiceGUI harness app you can view in the browser, plus a Playwright-based agent test script that spins up the harness, navigates it, and verifies behavior. You use the harness visually; the agent uses Playwright programmatically.

Pros

  • Fast visual feedback for you
  • Agent can drive and verify autonomously
  • One harness serves both purposes
  • Harness can expose a /test?class=SettingsNode.example URL pattern so agent can target specific settings classes
  • No full app or graph needed

Cons

  • Slightly more upfront design than A alone
  • Playwright must be installed as a dev dependency
