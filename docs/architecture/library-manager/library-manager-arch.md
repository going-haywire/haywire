# Library Manager â€” Architecture

## 1. Mental model

The Library Manager is a UI layer in `haywire-studio` that wraps two things behind a single page:

- **uv/pip** â€” the package layer that installs Python distributions.
- **The [Library System](../library-system/library-system-arch.md)** â€” the framework infrastructure that loads `Library` classes once they are installed.

Users see a curated *Available* list (libraries they could install), an *Enabled* list (libraries currently active), and a *Disabled* list. The data behind those lists comes from `marketplace.toml` files â€” declarative feeds that describe installable libraries, where to get them, and where to find their docs.

The Library Manager is *not* the Library System. It does not discover entry points or load classes â€” it issues `uv pip install` commands, then asks the Library System to rescan. Everything below the wrapper is the Library System's job.

| `share.py` | `haywire-studio/src/haywire_studio/share.py` | `haywire share` snippet generator |

### 2.2 The `marketplace.toml` format

A marketplace file is a TOML file with a `[[packages]]` array. Each entry describes one installable library.

```toml
[[packages]]
name        = "haybale-visiongraph"
version     = "0.0.1"
description = "Camera, video, and OpenCV nodes"
author      = "..."
source      = "git"
install_spec = "haybale-visiongraph @ git+https://github.com/user/repo.git#subdirectory=libraries/haybale-visiongraph"
tags        = ["vision", "camera", "video", "opencv"]
source_url  = "https://github.com/user/repo"
docs_url    = "https://raw.githubusercontent.com/user/repo/main/libraries/haybale-visiongraph/haybale_visiongraph/"
```

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | string | Yes | Pip distribution name (e.g. `haybale-visiongraph`) |
| `version` | string | Yes | Current version, display only |
| `description` | string | No | One-line human-readable description |
| `author` | string | No | Author name(s) |
| `source` | string | Yes | One of `"pypi"`, `"git"`, `"local"` |
| `install_spec` | string | Yes | Passed verbatim to `uv pip install` |
| `tags` | list of strings | No | Filter tags shown in the Library Manager |
| `source_url` | string | No | URL to the library's repository homepage |
| `docs_url` | string | No | URL or local path to the documentation directory |

### 2.3 The four `source` values

| `source` | `install_spec` example | When to use |
|---|---|---|
| `"local"` | `/abs/path/to/haybale-mylib` | Local editable installs (machine-specific; do not commit to a shared marketplace) |
| `"pypi"` | `haybale-visiongraph>=0.2.0` | Published packages on the Python Package Index |
| `"git"` (whole repo) | `git+https://github.com/u/r.git` | A package is the whole git repo |
| `"git"` (subdirectory) | `name @ git+https://github.com/u/r.git#subdirectory=path` | Monorepo with multiple libraries; `#subdirectory=` points to the folder containing `pyproject.toml` |

Local entries are generated by `haywire init --dev`. Git entries (subdir variant) are what `haywire share` emits.

### 2.4 The `[[sources]]` mechanism

The global `~/.haywire/marketplace.toml` does not contain `[[packages]]` entries directly â€” instead it lists `[[sources]]` that point to *other* marketplace files (typically remote URLs):

```toml
[[sources]]
name = "official"
url  = "https://haywire.dev/marketplace.toml"

[[sources]]
name = "my-team"
url  = "https://intranet.example.com/haywire/marketplace.toml"
```

Each `url` must point to a file with the `[[packages]]` format. The Library Manager fetches them asynchronously at page load, labels each entry with the source name, and merges them with the project-local marketplace into one Available list.

### 2.5 Where marketplace files live

| File | Path | Created by | Role |
|---|---|---|---|
| Project marketplace | `<workspace>/.haywire/marketplace.toml` | `haywire init` (always) | Primary feed; entries labelled `"project"` in UI |
| Global marketplace | `~/.haywire/marketplace.toml` | `ensure_global_config()` (first launch or `haywire init`) | `[[sources]]` list of remote feeds |

The project marketplace is editable in the UI via an inline TOML editor; the remote files referenced by global `[[sources]]` are not (you'd edit the local `~/.haywire/marketplace.toml` to add or remove the source).

### 3.2 Install pipeline

```text
User clicks Install on an Available entry
  â”‚
  â”śâ”€â”€ LibraryManager.install(entry) reads entry.install_spec
  â”śâ”€â”€ Subprocess: uv pip install <install_spec>
```

The install command `uv pip install` varies by source type:

| Source | Command run by `LibraryManager.install()` |
|---|---|
| `local` | `uv pip install -e /abs/path` |
| `pypi` | `uv pip install haybale-lib>=1.0.0` |
| `git` (whole) | `uv pip install git+https://github.com/u/r.git` |
| `git` (subdir) | `uv pip install name @ git+https://...#subdirectory=path` |

### 3.3 Install-type detection at runtime

After install, the Library System inspects each library's filesystem location and assigns an `InstallType` (`REGULAR`, `EDITABLE`, `FOLDER` â€” see [library-system Â§2.3](../library-system/library-system-arch.md#23-installtype-enum)). The Library Manager UI uses this to decide which actions are available:

| Action | `EDITABLE` | `REGULAR` | `FOLDER` |
|---|---|---|---|
| Save source (right panel) | Yes | No (read-only) | No (read-only) |
| Hot-reload | Yes | No | Yes |
| Disable / Enable | Yes | Yes | Yes |
| Uninstall via UI | Yes (`uv pip uninstall â€¦`) | Yes | Yes |

### 3.4 Documentation rendering

The right panel's Overview tab pulls library docs from one of two paths depending on whether the library is installed.

**Installed library â€” Overview tab.** Reads `<source_path>/OVERVIEW.md` from the filesystem (no network). `source_path` is the entry-point's resolved location.

**Installed library â€” Nodes / Widgets tab â†’ click â†’ Docs.** Reads `<source_path>/docs/nodes/<ClassName>.md` or `<source_path>/docs/widgets/<ClassName>.md`.

**Marketplace / Available library â€” center panel.** Uses `docs_url` from the marketplace entry, with this priority:

1. `docs_url` is a **local directory path** â†’ read `OVERVIEW.md` or `QUICKREF.md` from it.
2. `docs_url` is a **local file path** â†’ read it directly.
3. `docs_url` starts with `http` â†’ fetch the URL (appending `/OVERVIEW.md` if it ends with `/`).
4. `docs_url` empty â†’ heuristic GitHub raw URL derived from `source_url` / `install_spec`, trying `main` then `master`.
5. All else â†’ PyPI `long_description` fallback (for `source = "pypi"`).

### 3.5 Source code viewer

In the right panel's Source tab, the Library Manager uses `inspect.getfile(cls)` to locate the exact `.py` file for the selected component and displays it in a CodeMirror editor with Python syntax highlighting. For `EDITABLE` libraries, a Save button writes back; the [hot-reload](../hot-reload/hot-reload-arch.md) pipeline then picks up the change within ~0.5 s.

## 4. Boundary â€” what the Library Manager is not

- **Not a registry.** Class registries belong to the [Library System](../library-system/library-system-arch.md). The Library Manager *triggers* a rescan; it does not own the registry state.
- **Not a marketplace publisher.** It consumes `marketplace.toml`. Producing one for a published library is the job of `haywire share` â€” see [components/haybale-package](../../components/haybale-package/haybale-package-canon.md).
- **Not a build tool.** It does not build wheels or run `uv build` / `uv publish`.

## 5. Examples

### 5.1 Installing from a project-local source

A project's `<workspace>/.haywire/marketplace.toml`:

```toml
[[packages]]
name        = "haybale-mylib"
version     = "0.1.0"
description = "Local development library"
source      = "local"
install_spec = "/abs/path/to/myproject/libs/haybale-mylib"
docs_url     = "/abs/path/to/myproject/libs/haybale-mylib/haybale_mylib"
```

Click Install in the UI â†’ `uv pip install -e /abs/path/...` runs â†’ entry point appears â†’ Library System rescan loads it as `EDITABLE` â†’ Save source and hot-reload available.

### 5.2 Adding a remote feed

Edit `~/.haywire/marketplace.toml`:

```toml
[[sources]]
name = "official"
url  = "https://haywire.dev/marketplace.toml"
```

On next page load: the Library Manager fetches that URL asynchronously, parses its `[[packages]]`, and merges them into the Available list with the source label `official`.

### 5.3 What `haywire share` produces

```bash
uv run haywire share libs/haybale-mylib
```

Output (paste into any `marketplace.toml`):

```toml
[[packages]]
name         = "haybale-mylib"
version      = "0.1.0"
description  = "My custom nodes"
author       = "Your Name"
source       = "git"
install_spec = "haybale-mylib @ git+https://github.com/user/repo.git#subdirectory=libs/haybale-mylib"
tags         = []
source_url   = "https://github.com/user/repo"
docs_url     = "https://raw.githubusercontent.com/user/repo/main/libs/haybale-mylib/haybale_mylib/"
```

`share` handles SSHâ†’HTTPS conversion, computes the `#subdirectory=` fragment from the library's path relative to the git root, and derives `docs_url` from the GitHub raw content URL.

## 6. Open questions

- **Lazy library loading.** Currently every discovered library is loaded eagerly at startup (see [library-system Â§4.2](../library-system/library-system-arch.md#42-module-resolution-cost)). Could the Library Manager surface a Disable-by-default option for heavy libraries to defer loading?
- **Cross-repo dependency resolution.** A `marketplace.toml` entry can pin a Git tag, but conflicting transitive dependencies between two installed libraries are uv's problem, not the Library Manager's. Is there a UX layer that should surface those conflicts?
- **Documentation generation.** `OVERVIEW.md` / `QUICKREF.md` / `docs/nodes/<ClassName>.md` are produced by a Claude Code skill (`/docs`). Whether to ship a non-Claude generator (CLI-only) is undecided.
