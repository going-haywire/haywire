## Content-type legend

Pick one type per file in the **Type** column. Multiple types acceptable as `glossary+reference` if a doc genuinely is two things — but resist; usually one dominant type wins. `unclear` is fine when the dominant type isn't obvious from a quick read.

| Slug | Description |
|---|---|
| **`glossary`** | Term → definition tables; canonical vocabulary. |
| **`whitepaper`** | Foundational spec / vision document. Conceptual, version-numbered, big TOC. |
| **`architecture-desc`** | Component-by-component architectural description with file/class references. |
| **`architecture-tour`** | End-to-end narrative of how the system fits together (high-altitude). |
| **`impl-spec`** | Numbered implementation specification with mental model, invariants, lifecycle. |
| **`speculative-spec`** | Status: speculative. Describes a not-yet-implemented design. |
| **`conceptual-analysis`** | Grounded mental model of how an existing area actually works; no code changes proposed. |
| **`authors-guide`** | End-to-end task-oriented guide ('build a panel', 'create a library'). Long-form, multi-section. |
| **`tutorial`** | Code-first walkthrough showing how to do one thing. |
| **`cookbook`** | Short recipe = problem + minimal code, repeated. |
| **`api-reference`** | Signatures, enums, parameters. Lookup table for an API surface. |
| **`design-guide`** | Visual / UX rules, do/don't, tokens, conventions. |
| **`diagram`** | Primarily a diagram (ASCII or mermaid) with annotations. |
| **`quick-reference`** | Cheat sheet / TL;DR for an existing system. |
| **`changelog-style`** | What changed / what was added / migration notes. |
| **`test-notes`** | How tests are organised, harness usage, fixtures. |
| **`unclear`** | Couldn't categorise from a quick scan — needs another look. |

## Suggested workflow

1. **Type column first.** Skim each doc, pick the dominant content type from the legend. Don't agonise — `unclear` is a valid temporary value.
2. **Re-sort by type** (your editor's column-sort works on Markdown tables) to see what kinds of content the corpus actually has. This will surface gaps and duplicates.
3. **Map to STRUCTURE.md.** Every type maps to one or more homes:
   - `glossary` → `reference/glossary.md`
   - `whitepaper` → `archive/whitepaper/`
   - `architecture-desc` / `architecture-tour` / `impl-spec` → `architecture/<area>/`
   - `speculative-spec` → `internals/speculative/` (stays out of `docs/`) or promote when implemented
   - `conceptual-analysis` → usually feeds `architecture/<area>/01-overview.md`
   - `authors-guide` / `tutorial` / `cookbook` → `components/<area>/`
   - `api-reference` → final chapter of the relevant story (`NN-reference.md`) or `reference/api/`
   - `design-guide` → `reference/design-guide.md`
   - `diagram` → embed inside the relevant story; rarely a standalone home
   - `quick-reference` → final chapter (`quick-reference.md`) of the relevant story
   - `changelog-style` → usually drop after migration; signal of historical drift
   - `test-notes` → `architecture/<area>/NN-testing.md` or general contributor docs
4. **Hand back** — once types are filled in, the next pass assigns each survivor to a destination folder under `docs/`.
